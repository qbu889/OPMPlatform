# 部署脚本优化说明

## 🚀 性能优化对比

### 优化前（逐个文件上传）
```bash
# 假设有100个后端文件变更
for file in ${BACKEND_FILES[@]}; do
    scp "$file" remote:/path/    # 每个文件都要建立SSH连接、校验、传输
done
# 耗时：约 5-10 分钟（取决于文件数量和网络）
```

**问题：**
- ❌ 每个文件都要单独建立SSH连接
- ❌ 每个文件都要单独校验完整性
- ❌ 大量小文件传输效率极低
- ❌ 网络延迟累积效应明显

---

### 优化后（打包批量上传）

#### 1️⃣ 后端文件：打包压缩后一次性上传

```bash
# 步骤1: 本地打包
tar -czf backend_update.tar.gz file1.py file2.py routes/*.py ...

# 步骤2: 一次性上传压缩包
scp backend_update.tar.gz remote:/tmp/

# 步骤3: 远程解压
ssh remote "cd /project && tar -xzf /tmp/backend_update.tar.gz"
```

**优势：**
- ✅ 只需建立1次SSH连接
- ✅ 只需1次文件传输（压缩后更小）
- ✅ 只需1次完整性校验
- ✅ 适合大量小文件场景
- ⏱️ **耗时：约 30秒-1分钟**（提升5-10倍）

---

#### 2️⃣ 前端文件：智能选择传输方式

```bash
# 优先使用rsync（支持增量同步）
if command -v rsync &> /dev/null; then
    rsync -avz --delete dist/ remote:/path/frontend/dist/
else
    # 降级使用scp
    scp -r dist/* remote:/path/frontend/dist/
fi
```

**rsync优势：**
- ✅ 只传输变更的文件块（增量同步）
- ✅ 自动删除远程多余文件（--delete）
- ✅ 支持断点续传
- ✅ 压缩传输（-z参数）
- ⏱️ **二次部署耗时：约 5-10秒**（几乎瞬间完成）

## 🔧 技术细节

### 1. 后端打包策略

```bash
# 创建临时目录
TEMP_DIR=$(mktemp -d)

# 保持目录结构复制文件
for file in "${BACKEND_FILES[@]}"; do
    DEST_DIR="$TEMP_DIR/$(dirname $file)"
    mkdir -p "$DEST_DIR"
    cp "$file" "$DEST_DIR/"
done

# 打包并清理源文件
cd "$TEMP_DIR"
tar -czf backend_update.tar.gz --remove-files $(find . -type f)
```

**关键点：**
- `mktemp -d`：创建安全的临时目录
- `--remove-files`：打包后立即删除源文件，节省空间
- `find . -type f`：递归查找所有文件
- `sed 's|^\./||'`：去除路径前缀 `./`

---

### 2. 前端rsync增量同步

```bash
rsync -avz --delete dist/ remote:/path/frontend/dist/
```

**参数说明：**
- `-a`：归档模式（保留权限、时间戳等）
- `-v`：显示详细输出
- `-z`：传输时压缩
- `--delete`：删除目标端多余文件（保证一致性）
- `dist/`：末尾的 `/` 表示同步目录内容而非目录本身

---

### 3. 容错处理

```bash
# 检查rsync是否可用
if command -v rsync &> /dev/null; then
    # 使用rsync
else
    # 降级到scp
fi

# 清理临时文件
rm -rf "$TEMP_DIR"
```

## ⚠️ 注意事项

### 1. 大文件处理

如果单个文件超过100MB，建议：
- 使用 `split` 命令分割文件
- 或使用专门的传输工具（如 `lftp`）

### 2. 并发控制

当前脚本是串行执行，如果需要更快：
```bash
# 并行上传多个压缩包（谨慎使用）
scp part1.tar.gz remote:/tmp/ &
scp part2.tar.gz remote:/tmp/ &
wait
```

### 3. 磁盘空间

确保远程服务器有足够空间：
```bash
ssh root@8.146.228.47 "df -h /tmp"
ssh root@8.146.228.47 "du -sh /project/backups"
```
