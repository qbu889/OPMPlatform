## 1. 功能概述

### 1.1 背景

当前系统已实现排班信息的钉钉定时推送功能，但配置较为固定，缺乏灵活性。需要抽象出一个通用的钉钉推送平台，支持：

- **多业务场景**: 排班推送、告警通知、日报周报、会议提醒等
- **灵活配置**: 自定义推送内容、时间、接收人
- **可视化预览**: 实时查看推送效果
- **智能调度**: 支持复杂的时间规则和条件触发

### 1.2 目标用户

| 角色 | 使用场景 | 核心需求 |
|------|----------|----------|
| 运维管理员 | 排班信息推送 | 定时推送、人员配置、时段选择 |
| 团队负责人 | 工作日报/周报 | 模板化内容、自动汇总、定时发送 |
| 项目经理 | 项目进度通知 | 条件触发、@特定人员、附件支持 |
| 系统管理员 | 告警通知 | 实时监控、紧急推送、多渠道备份 |

### 1.3 核心价值

1. **统一推送平台**: 集中管理所有钉钉推送任务
2. **配置可视化**: 无需代码即可配置推送规则
3. **效果可预览**: 发送前预览消息样式和内容
4. **灵活可扩展**: 支持自定义数据源和消息模板

## 3. 技术架构

### 3.1 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | Vue 3 + Element Plus | 单页应用 |
| 后端框架 | Flask + Blueprint | RESTful API |
| 定时调度 | APScheduler | Python 定时任务 |
| 数据库 | MySQL 8.0 | 配置和数据存储 |
| 缓存 | Redis (可选) | 数据缓存、队列 |
| 模板引擎 | Jinja2 | 消息模板渲染 |
| 加密 | cryptography (Fernet) | Webhook 加密 |
| HTTP 客户端 | requests | 钉钉 API 调用 |

### 3.3 数据流转

```
1. 用户配置推送任务
   ↓
2. 配置保存到 MySQL (Webhook 加密存储)
   ↓
3. APScheduler 注册定时任务
   ↓
4. 到达推送时间，触发任务
   ↓
5. 从数据源获取数据 (API/DB/静态)
   ↓
6. Jinja2 渲染消息模板
   ↓
7. 构建钉钉消息 JSON
   ↓
8. 发送 HTTP POST 到钉钉 Webhook
   ↓
9. 记录推送结果到数据库
   ↓
10. 失败则重试（最多 3 次）
```

### 4.2 表结构定义

#### 4.2.1 推送配置表 (dingtalk_push_config)

```sql
CREATE TABLE dingtalk_push_config (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '配置ID',
    name VARCHAR(100) NOT NULL COMMENT '任务名称',
    description VARCHAR(500) COMMENT '任务描述',
    
    -- Webhook 配置
    webhook_url TEXT NOT NULL COMMENT '加密后的 Webhook URL',
    secret_key VARCHAR(200) COMMENT '钉钉加签密钥（可选）',
    
    -- 推送对象
    at_mobiles JSON COMMENT '@指定人员手机号列表 ["13800138000"]',
    at_all BOOLEAN DEFAULT FALSE COMMENT '是否@所有人',
    
    -- 消息配置
    message_type VARCHAR(20) NOT NULL DEFAULT 'markdown' COMMENT '消息类型: markdown/actionCard/text',
    template_content TEXT NOT NULL COMMENT '消息模板内容（Jinja2 模板）',
    data_source_config JSON COMMENT '数据源配置 {type: "api/sql/static", config: {...}}',
    
    -- 调度配置
    schedule_config JSON NOT NULL COMMENT '调度配置 {type: "cron/daily/weekly", config: {...}}',
    timezone VARCHAR(50) DEFAULT 'Asia/Shanghai' COMMENT '时区',
    
    -- 执行配置
    enabled BOOLEAN DEFAULT TRUE COMMENT '是否启用',
    category VARCHAR(50) DEFAULT 'general' COMMENT '分类: roster/alert/report',
    priority INT DEFAULT 0 COMMENT '优先级 (0-10, 越大越优先)',
    max_retries INT DEFAULT 3 COMMENT '最大重试次数',
    timeout_seconds INT DEFAULT 10 COMMENT '请求超时时间（秒）',
    
    -- 审计字段
    created_by VARCHAR(50) COMMENT '创建人',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    
    INDEX idx_enabled (enabled),
    INDEX idx_category (category),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='钉钉推送配置表';
```

#### 4.2.2 推送历史表 (dingtalk_push_history)

```sql
CREATE TABLE dingtalk_push_history (
    id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '历史记录ID',
    config_id INT NOT NULL COMMENT '配置ID',
    
    -- 执行信息
    trigger_time DATETIME NOT NULL COMMENT '触发时间',
    start_time DATETIME COMMENT '开始执行时间',
    end_time DATETIME COMMENT '结束执行时间',
    execution_duration_ms INT COMMENT '执行耗时（毫秒）',
    
    -- 执行结果
    status VARCHAR(20) NOT NULL COMMENT '状态: success/failed/retrying',
    response_data JSON COMMENT '钉钉 API 响应数据',
    error_message TEXT COMMENT '错误信息',
    retry_count INT DEFAULT 0 COMMENT '重试次数',
    
    -- 消息内容快照
    message_snapshot JSON COMMENT '实际发送的消息内容',
    
    -- 审计字段
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '记录时间',
    
    FOREIGN KEY (config_id) REFERENCES dingtalk_push_config(id) ON DELETE CASCADE,
    INDEX idx_config_id (config_id),
    INDEX idx_trigger_time (trigger_time),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='钉钉推送历史表';
```

#### 4.2.3 推送日志表 (dingtalk_push_log)

```sql
CREATE TABLE dingtalk_push_log (

    INDEX idx_config_id (config_id),
    INDEX idx_history_id (history_id),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='钉钉推送日志表';
```

### 7.2 节假日过滤

```python
def should_execute(schedule_config, current_time):
    """判断是否应该执行推送"""
    
    # 检查工作日过滤
    if schedule_config.get('exclude_holidays'):
        if is_holiday(current_time.date()):
            return False
    
    # 检查星期过滤
    if 'weekdays' in schedule_config:
        if current_time.weekday() + 1 not in schedule_config['weekdays']:
            return False
    
    return True
```

### 7.3 失败重试策略

```python
class RetryStrategy:
    def __init__(self, max_retries=3, base_delay=1):
        self.max_retries = max_retries
        self.base_delay = base_delay
    
    def get_delay(self, retry_count):
        """指数退避延迟"""
        return min(self.base_delay * (2 ** retry_count), 60)
    
    def should_retry(self, retry_count, error):
        """判断是否应该重试"""
        if retry_count >= self.max_retries:
            return False
        
        # 某些错误不重试（如配置错误）
        if isinstance(error, ConfigurationError):
            return False
        
        return True
```

### 8.3 模板安全性

- **沙箱环境**: 限制模板中可执行的函数
- **白名单**: 只允许安全的过滤器和函数
- **超时控制**: 模板渲染超时限制（5秒）
- **资源限制**: 限制循环次数和递归深度

## 9. 安全与权限

### 9.1 Webhook 加密

```python
from cryptography.fernet import Fernet

# 加密
def encrypt_webhook(url):
    key = os.environ.get('WEBHOOK_ENCRYPTION_KEY')
    fernet = Fernet(key.encode())
    return fernet.encrypt(url.encode()).decode()

# 解密
def decrypt_webhook(encrypted_url):
    key = os.environ.get('WEBHOOK_ENCRYPTION_KEY')
    fernet = Fernet(key.encode())
    return fernet.decrypt(encrypted_url.encode()).decode()
```

## 10. 监控与日志

### 10.1 日志级别

| 级别 | 使用场景 |
|------|----------|
| DEBUG | 详细的调试信息 |
| INFO | 推送开始/结束、成功信息 |
| WARNING | 重试、超时警告 |
| ERROR | 推送失败、配置错误 |
| CRITICAL | 系统级错误 |

### 10.2 日志示例

```
2026-04-21 08:00:00 [INFO] 开始执行推送任务: 每日排班推送 (ID: 1)
2026-04-21 08:00:00 [INFO] 获取数据源成功: API 返回 2 条记录
2026-04-21 08:00:00 [INFO] 模板渲染完成: 生成 256 字节消息
2026-04-21 08:00:00 [INFO] 发送 POST 请求到钉钉 Webhook
2026-04-21 08:00:01 [INFO] 推送成功: errcode=0, errmsg=ok
2026-04-21 08:00:01 [INFO] 推送任务完成: 耗时 250ms
```

### 10.3 告警规则

- **连续失败**: 同一任务连续失败 3 次，发送告警
- **成功率下降**: 成功率低于 80%，发送告警
- **执行超时**: 单次执行超过 30 秒，记录警告
- **队列积压**: 待执行任务超过 100 个，发送告警

### 10.4 监控指标

| 指标 | 说明 |
|------|------|
| 推送总量 | 累计推送次数 |
| 成功率 | 成功次数 / 总次数 |
| 平均耗时 | 平均执行时间 |
| P95 耗时 | 95% 的请求耗时 |
| 失败原因分布 | 各类错误的占比 |
| 活跃任务数 | 启用的任务数量 |
