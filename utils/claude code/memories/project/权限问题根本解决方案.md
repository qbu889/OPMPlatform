# 权限问题根本解决方案

## 问题背景

Nginx 500/403 错误反复出现,原因是项目文件所有权不统一:
- `/project/wordToWord/` 所有者可能是 `501:games`(从 macOS 上传)
- SCP 上传的文件所有者是 `root`
- Nginx worker 进程以 `www` 用户运行,需要访问所有文件

## 根本原因

每次部署时:
1. 使用 `scp` 上传文件,所有者变成 `root`
2. 原始项目文件所有者是 `501:games`
3. 只修复了部分目录权限,没有统一整个项目
4. 导致 Nginx 无法访问某些路径

## 解决方案

### 1. 统一项目所有权为 www:www

**核心原则**: 所有项目文件的所有者统一为 `www:www`,这样:
- Nginx worker (www) 可以正常访问静态文件
- Python 后端服务也能正常读写
- 避免权限不一致导致的 403 错误

### 2. 使用统一的权限修复脚本

**脚本位置**: `scripts/fix_permissions.sh`

**功能**:
```bash
#!/bin/bash
# 1. 统一所有者为 www:www
chown -R www:www /project/wordToWord

# 2. 设置目录权限 (755)
find /project/wordToWord -type d -exec chmod 755 {} \;

# 3. 设置文件权限 (644)
find /project/wordToWord -type f -exec chmod 644 {} \;

# 4. 设置脚本可执行权限
chmod +x /project/wordToWord/*.sh
chmod +x /project/wordToWord/scripts/*.sh
```

### 3. 集成到部署流程

`deploy.sh` 已更新,在每次部署后自动执行权限修复:

```bash
# 步骤 7: 统一修复项目权限(根本解决方案)
if [ -f "scripts/fix_permissions.sh" ]; then
    bash scripts/fix_permissions.sh
else
    # 降级方案:直接修复权限
    chown -R www:www /project/wordToWord
    find /project/wordToWord -type d -exec chmod 755 {} \;
    find /project/wordToWord -type f -exec chmod 644 {} \;
fi
```

## 权限规范

| 类型 | 权限 | 说明 |
|------|------|------|
| 目录 | 755 (rwxr-xr-x) | 所有者可读写执行,其他人可读执行 |
| 普通文件 | 644 (rw-r--r--) | 所有者可读写,其他人只读 |
| 脚本文件 | 755 (rwxr-xr-x) | 可执行的 shell 脚本 |
| 所有者 | www:www | 统一为 Nginx 用户 |

## 验证方法

```bash
# 1. 检查项目根目录权限
ls -la /project/ | grep wordToWord
# 应该显示: drwxr-xr-x www www

# 2. 检查 frontend 目录权限
ls -la /project/wordToWord/ | grep frontend
# 应该显示: drwxr-xr-x www www

# 3. 检查 dist 目录权限
ls -la /project/wordToWord/frontend/ | grep dist
# 应该显示: drwxr-xr-x www www

# 4. 测试 Nginx 访问

```

## 常见问题

### Q: 为什么不用 root:root?
A: Nginx worker 进程以 www 用户运行,如果文件所有者是 root,需要给其他用户开放权限,不够安全。统一为 www:www 更符合最小权限原则。

### Q: Python 后端用 root 运行会有问题吗?
A: 不会。root 用户可以访问 www 用户的文件(因为目录有执行权限,文件有读权限)。但建议后端也用 www 用户运行更安全。

### Q: 上传新文件后需要重新修复权限吗?
A: 是的。`deploy.sh` 已集成自动修复,每次部署都会执行。如果手动上传文件,需要手动执行修复脚本。

### Q: 为什么之前只修复 frontend/dist?
A: 这是临时方案,治标不治本。Nginx 需要访问所有父目录才能到达子目录,所以必须统一整个项目的权限。

## 历史教训

这个问题出现过多次,原因是:
1. ❌ 只修复子目录,忽略父目录权限
2. ❌ 使用不同的所有者(root/www/501)
3. ❌ 没有统一的权限管理策略

现在的方案:
1. ✅ 统一整个项目为 www:www
2. ✅ 自动化修复,集成到部署流程
3. ✅ 标准化权限规范(755/644)
