# 排班系统改进说明

## 问题分析

### 原问题
1. **日常排班问题**: 郑晨昊在 2026-02-02 排班后，2026-02-03、2026-02-06 又连续排班，没有正确轮换
2. **节假日排班问题**: 郑晨昊在 2026-02-08 排班后，2026-02-15 又排班，间隔 7 天不符合轮换规则

### 根本原因
1. 每次排班后没有正确更新轮换索引
2. 没有考虑前一天已排班人员的延后处理
3. 人员选择逻辑不够智能，无法过滤已排班人员

## 改进方案

### 1. 数据库表结构增强
为 `roster` 表添加以下字段:
- `is_main`: 标识是否为主班
- `rotation_index`: 记录轮换索引
- `remark`: 备注信息 (如请假调整说明)
- `created_at`: 创建时间戳

执行迁移脚本:
```bash
mysql -u root -p schedule < routes/排班/alter_roster_table.sql
```

### 2. 核心排班逻辑改进

#### 2.1 动态人员配置
- 从 `staff_config` 表动态读取核心人员和测试人员
- 支持通过 API 实时配置人员

#### 2.2 智能轮换机制
```python
def _select_staff_from_queue(self, slot_type, available_staffs, exclude_staffs=None):
    """
    从轮换队列中选择人员
    1. 从当前索引开始查找
    2. 过滤请假和已排班人员
    3. 返回选中人员和新的索引位置
    """
```

#### 2.3 前一天排班延后处理
```python
def _get_prev_day_staff(self, target_date, time_slot_pattern):
    """获取前一天某时段的排班人员，今天应该延后"""
```

**规则**:
- 如果前一天 8:00-12:00 或 18:00-21:00 排了某人，今天该时段应跳过此人
- 延后到轮换队列的下一位

#### 2.4 请假处理优化
- 全天请假：过滤所有时段
- 时段请假：只过滤对应时段
- 无人可排时的默认处理：安排核心人员并添加备注

### 3. 轮换队列更新规则

#### 日常排班 (以 5 人为例)
```
初始队列：[郑晨昊，林子旺，曾婷婷，陈伟强，吴绍烨]
索引：0

第 1 天：
- 选择：郑晨昊 (索引 0)
- 更新索引：1
- 剩余可排：[林子旺，曾婷婷，陈伟强，吴绍烨]

第 2 天:
- 从索引 1 开始：林子旺
- 更新索引：2
- 剩余可排：[曾婷婷，陈伟强，吴绍烨，郑晨昊]

第 3 天:
- 从索引 2 开始：曾婷婷
- 以此类推...
```

#### 节假日排班
```
初始队列：[郑晨昊，林子旺，曾婷婷，陈伟强，吴绍烨]
索引：0

第 1 个节假日:
- 选择：郑晨昊 (索引 0)
- 更新索引：1
- 剩余可排：[林子旺，曾婷婷，陈伟强，吴绍烨]

第 2 个节假日:
- 从索引 1 开始：林子旺
- 更新索引：2
- 剩余可排：[曾婷婷，陈伟强，吴绍烨，郑晨昊]

第 3 个节假日:
- 从索引 2 开始：曾婷婷
- 以此类推...
```

### 4. 特殊场景处理

#### 场景 1: 有人请假
```
队列：[A, B, C, D, E]
索引：0
B 请假

选择过程:
1. 检查 A(索引 0): 可用 → 选 A
2. 更新索引：1
下次从 B 开始，但 B 请假，跳过→选 C
```

#### 场景 2: 前一天早班人员今天延后
```
昨天早班：A
今天队列：[A, B, C, D, E]
索引：0

选择过程:
1. 检查 A(索引 0): 昨天排过→跳过
2. 检查 B(索引 1): 可用→选 B
3. 更新索引：2
```

#### 场景 3: 所有人都请假
```
队列：[A, B, C, D, E]
全部请假

处理:
- 默认排核心人员
- 添加备注："因当前无可排人员默认排核心人员"
```

## 常见问题

### Q1: 为什么郑晨昊频繁排班？
A: 之前的逻辑没有正确更新轮换索引，每次都从队列开头选人。

### Q2: 如何处理请假？
A: 系统会自动过滤请假人员，从轮换队列中选择下一个可用人员。

### Q3: 节假日和日常排班是否独立？
A: 是的，各自维护独立的轮换队列和索引。

### Q4: 前一天排过的人今天还会排吗？
A: 对于 8:00-9:00 和 18:00-21:00 时段，会避开前一天同时段的人员。
